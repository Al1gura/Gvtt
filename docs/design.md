## 〇、产品五维度（核心指导思想）

Gvvt 的**所有功能都是从这五个维度推导出来的**。任何功能讨论都必须回答它服务于哪几个维度。

| # | 维度 | 一句话 | 受众 |
|---|------|--------|------|
| ① | **场景呈现** | 玩家能看到立体的、空间上可信的场景 | 玩家 |
| ② | **场景因果** | 场景变化是真实的——因果关系，不是假装 | 玩家 |
| ③ | **模组管理** | GM 管理的是一个完整冒险，不是一张张孤立的地图 | GM |
| ④ | **交互逻辑** | 跑团中软件不打断游戏——GM 一气呵成 | GM |
| ⑤ | **准备成本** | GM 备团的投入可控——不被工具拖累 | GM |

**画面品质不是第六个维度**——它是所有维度的底线约束，不是目标方向。

### 五维度独立性验证

- ①呈现 vs ②因果：静态呈现 vs 动态变化真实性。2D VTT 能做①但永远做不到②
- ②因果 vs ④交互：④是 GM 能执行操作，②是操作后世界真的变了
- ③模组 vs ⑤成本：管理能力 vs 时间投入，不同维度
- ①呈现 vs ⑤成本：GM 想给玩家看的 vs 为这个要花多少时间——核心张力
- ④交互 vs ⑤成本：直觉操作 vs 操作耗时，不是同一件事

### MVP 开发顺序

| 顺序 | 功能 | 理由 |
|------|------|------|
| P0 | ① 3D 俯视地图展示 | 地基，先验证 Godot 渲染可行 |
| P1 | ② 地面绘制 / 网格编辑 | 编辑态基础 |
| P1 | ③ 规则运行 | Gvvt 的交互载体 |
| P2 | ④ 动态光照 + LOS | 核心卖点，依赖墙/光源物件 |
| P2 | ⑤ 特效系统 | 依赖光照系统 |
| P3 | ⑥ 投屏模式 | 其他做完后才有意义 |

## 一、产品定位

不是游戏，是一个**用 Godot 引擎做的 GM 桌面工具**——开源的 3D 俯视交互地图引擎。

| 维度 | 结论 |
|------|------|
| 首要场景 | 线下跑团，GM 电脑操作，投屏给所有人看 |
| 终端形态 | 单 exe，GM 一人操作，无需玩家端 |
| 线上预留 | 保留接口，初期不做联网功能 |
| 明确不做 | 联网同步、骰子系统、规则系统、模组市场、玩家端、账号 |
| 许可证 | MIT |

---

## 二、核心理念

**「一张活的立体地形桌」**

- 俯视正交 3D 地形（像看实体桌面）
- 实时光照 + 视线遮挡
- 地图可交互（墙可破坏、地形可改变、光照联动）
- 2D 立绘 Token 站在 3D 地形上
- 双模式：线下投屏（实物棋子）+ 网团（电子 Token）
- **锦上添花，不是推翻重来**——实体跑团已经很好了，电子化去掉计算摩擦、增加视觉沉浸

## 三、模式架构

```
一体化的单 exe 应用
    │
    ├── 编辑态（备团/搭地图）
    │   ├── 地面绘制（笔刷铺材质/颜色）
    │   ├── 规则运行（放置墙/土丘/火把/Token 等带交互属性的物件）
    │   ├── 属性标记（可破坏/可交互/光源等游戏数据）
    │   └── 场景管理（保存/加载/新建）
    │
    └── 运行态（跑团中）
        ├── Token 拖拽
        ├── 光源开关
        ├── 特效触发
        ├── 墙体破坏 + LOS 实时联动
        └── 投屏模式（全屏无 UI）
```

- 编辑态 ↔ 运行态 **一键无缝切换**，类似 Unity 的 Editor ↔ Play 模式
- 切换时不丢失场景状态（Token 位置、光照状态等保留）

## 四、MVP 功能清单（6 项）

### 4.1 3D 俯视地图展示
- 正交相机，自由缩放/平移
- 网格渲染（可开关）

### 4.2 地面绘制 / 网格编辑
- 笔刷：在 GridMap 格子上填材质/颜色，铺地面底色
- 地形升降（高度调整）
- 水体/路径笔刷

### 4.3 规则运行（编辑态放置 + 运行态交互）
- 墙体（可选标记：可破坏）
- 地形块（土丘/岩石/台阶）
- 植被（树/灌木）
- 建筑预制件
- 光源物件（火把/篝火/魔法光，可标记：可开关）
- Token 标记（角色/怪物）

### 4.4 动态光照 + 视线遮挡（LOS）
- 环境光源（环境光调节）
- 区域光源（火炬/篝火/法术光，可设置颜色/范围/亮度）
- 实时阴影
- 墙体 LOS 遮挡 + 墙破坏后光照从缺口穿过

### 4.5 特效系统
- 预设法术 VFX（火球/闪电/治疗）
- 粒子效果（雨/雪/火焰/烟雾）
- 运行态手动触发

### 4.6 投屏模式
- 一键全屏输出到第二屏，无 UI
- 仅显示 3D 场景

## 五、编辑方式（核心设计决策）

### 5.1 地面/地形编辑
- **笔刷制**：在网格上拖拽绘制材质/颜色/高度
- 不是逐格点放

### 5.2 规则运行编辑
- **物件仓库**：左侧面板分类显示可用物件（地形/建筑/植被/光源/Token）
- **拖拽放置**：选中物件，拖到目标格子放下，同时设定该物件的游戏属性（可破坏/可交互/光源等）

### 5.3 为什么是笔刷 + 物件两套（不是纯物件）
- 地面底色和纹理是"连续区域"，笔刷一次覆盖比逐格放物件高效
- 功能性元素（墙/火把/Token）是"离散个体"，物件放置 + 属性标记更合理
- 两者不是替代关系，是不同层面的工具

### 5.4 不做什么
- 不做程序化纹理混合
- 不做智能边缘过渡（笔刷之间硬边，通过物件装饰掩盖过渡）

## 六、技术选型

| 项目 | 选型 | 说明 |
|------|------|------|
| 引擎 | Godot 4.4+ | MIT 协议，Vulkan 原生渲染 |
| 地形破坏 | Godot-Destruction | 开源，实时 Mesh 碎裂 + 穿孔 |
| 特效 | VFEZ-godot + GPUParticles3D + 商业 VFX 包 | 粒子系统原生支持 |
| 3D 资产 | AI 生成（Meshy/Rodin/Luma）+ 人工审核 | 物件模型来源 |
| 存档格式 | 待定 | JSON / Godot Resource |

## 七、场景管理

- 每个战斗地图是一个独立场景文件
- GM 可以在场景之间切换
- 场景切换时保留 Token 状态（位置、光开关状态、破坏状态）
- 内置若干预设场景模板（后续可扩展）
